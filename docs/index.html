Satellite Change Detection

A hands-on tutorial. Map deforestation, mining, urbanisation, floods, burns and surface-water change anywhere on Earth from free satellite data — then export georeferenced GeoTIFFs and print-ready maps. All in Python, powered by Google Earth Engine.

Python 3.11+ Google Earth Engine Sentinel-1 & Sentinel-2 Free & open data MIT licensed

1 · What you'll build

By the end you'll run a single command like this…

python3 detect.py -s deforestation --lat -3.333 --lon 122.25 --radius 6 --map

…and get three things for your chosen area:

  • a quick-look PNG of the change,
  • a full-resolution, georeferenced GeoTIFF you can open in QGIS/ArcGIS, and
  • a value-added map (A4 PDF + PNG) with legend, scale bar, statistics and a location inset.

You pick a scenario (what kind of change) and a location (a coordinate). The tool selects the right remote-sensing method automatically. No prior GIS coding required.

2 · How it works (the science)

“Change detection” compares the landscape at two (or more) points in time and highlights what changed. Different phenomena leave different spectral or radar fingerprints, so each scenario uses a purpose-built index or method:

ScenarioMethod & formulaSensorBasis
deforestationΔNDVI < 0. NDVI = (NIR−Red)/(NIR+Red) = (B8−B4)/(B8+B4). Vegetation loss drops NDVI.Sentinel-2NDVI, Rouse et al. 1973; Tucker 1979
urbanizationΔNDBI > 0. NDBI = (SWIR−NIR)/(SWIR+NIR) = (B11−B8)/(B11+B8). Built-up surfaces reflect more SWIR.Sentinel-2Zha, Gao & Ni 2003
burndNBR. NBR = (NIR−SWIR2)/(NIR+SWIR2) = (B8−B12)/(B8+B12); burns lower NBR.Sentinel-2Key & Benson 2006
waterΔNDWI. NDWI = (Green−NIR)/(Green+NIR) = (B3−B8)/(B3+B8).Sentinel-2McFeeters 1996
floodSAR water: smooth water gives low VV backscatter. Flood = water in event window but not in baseline.Sentinel-1UN-SPIDER Recommended Practice
miningSIRAD: mean VH backscatter of 3 periods → an R/G/B image (new activity = blue). Plus ΔNDVI loss.Sentinel-1 + S2Multi-temporal SAR compositing
Why radar for floods and mining? Optical sensors can't see through clouds — a serious problem during floods and in the wet tropics. Sentinel-1 radar penetrates cloud and works day or night, so it captures change when optical imagery is blind. VH polarisation is most sensitive to land-surface change; VV best separates smooth open water.

Two ways clouds are handled

Optical scenarios never rely on a single image. For each period the tool builds a cloud-masked median composite of many Sentinel-2 scenes: every scene is masked pixel-by-pixel using the Scene Classification (SCL) band, then the median is taken. The result is effectively cloud-free even in cloudy regions. The standalone Sentinel-2 downloader additionally prefers a single scene with ≤10% cloud, falling back to the composite when none exists.

3 · Setup

You need Python 3.11+, git, and a free Google Earth Engine account (sign up; approval is usually quick). The optional documentary-video pipeline additionally needs ffmpeg and an ElevenLabs key — you can ignore those for change detection.

  1. Clone the repository
    git clone https://github.com/firmanhadi21/rs-change-detection.git
cd rs-change-detection
  2. Install the Python dependencies
    pip install -r requirements.txt
    This pulls earthengine-api, rasterio, numpy, matplotlib and contextily (for maps).
  3. Authenticate Earth Engine (one time)
    earthengine authenticate
    Opens a browser to link your Google account. Alternatively drop a service-account key at scripts/config/ee-geodetic.json — the tools detect it automatically.
  4. Check it's working
    python3 detect.py --list
    Should print the six available scenarios.

4 · Your first detection

Let's map vegetation loss around a coordinate. Pick any lat/lon (decimal degrees; south and west are negative).

python3 detect.py -s deforestation --lat -3.333 --lon 122.25 --radius 6

What happens: the tool builds a square area of interest (half-side = radius km), makes a cloud-free Sentinel-2 NDVI composite for a baseline year and a recent year, subtracts them, and downloads the result. You'll see something like:

=== Change detection: deforestation ===
Location: -3.333, 122.25  radius 6.0 km  [m3p333_122p25]
GEE: service account …
Downloading dndvi PNG...   Saved: images/deforestation_dndvi_m3p333_122p25.png
Downloading dndvi GeoTIFF... Saved: data/deforestation_dndvi_m3p333_122p25.tif
=== Results ===
  "pct_affected": 6.3, "pct_severe": 2.7, "scenes_pre": 30, "scenes_post": 25 …
Negative latitude tip. A leading - can confuse the shell's argument parser if you use -l. Either use --lat/--lon (shown above) or attach the value with =: -l=-3.333,122.25.

5 · Reading the outputs

FileWhat it isOpen with
images/<scenario>_<product>_<name>.pngColoured quick-lookAny image viewer
data/<…>.tifFull-resolution, georeferenced GeoTIFF (EPSG:4326)QGIS, ArcGIS, rasterio
data/<scenario>_<name>_stats.jsonStatistics: % area affected, mean Δ, scene countsAny text editor

For optical scenarios the change layer is a diverging palette: red = loss, pale = little change, green = gain. The stats.json quantifies it — e.g. pct_affected is the share of the area whose change passed the threshold.

Thresholds are a starting point, not gospel. The default per-scenario thresholds (e.g. ΔNDVI < −0.15) are sensible defaults. As Key & Benson note for burn severity, cut-offs should ideally be calibrated with local ground truth. Treat the % figures as comparative, not absolute.

6 · Every scenario

List them any time with python3 detect.py --list. Commands:

# Deforestation / vegetation loss (default baseline vs recent year)
python3 detect.py -s deforestation --lat -3.333 --lon 122.25

# Mining — radar temporal (SIRAD) + NDVI loss, two products
python3 detect.py -s mining --lat -3.333 --lon 122.25

# Urbanisation — built-up (NDBI) gain
python3 detect.py -s urbanization --lat -6.2 --lon 106.8 --radius 12

# Surface-water change (NDWI)
python3 detect.py -s water --lat -0.5 --lon 117.1

# Flood — needs an explicit baseline and event window
python3 detect.py -s flood --lat 27.2 --lon 68.3 \
    --pre 2022-07-01:2022-07-25 --post 2022-08-20:2022-09-10

# Burn severity — needs pre-fire and post-fire windows
python3 detect.py -s burn --lat -7.5 --lon 110.4 \
    --pre 2025-08-01:2025-08-20 --post 2025-09-10:2025-09-30
Flood and burn require dates. They compare a specific event to a baseline, so you must pass --pre START:END --post START:END (format YYYY-MM-DD:YYYY-MM-DD). The other scenarios have sensible default windows you can still override.

7 · Make a map product

Add --map to any run to render an A4-landscape map sheet (PDF + PNG) per product: OpenStreetMap basemap, the change layer, title, legend, a statistics panel, a location inset, coordinate grid, scale bar and north arrow.

python3 detect.py -s mining --lat -3.333 --lon 122.25 --map

Already have results and just want to re-render (no Earth Engine needed)? Every run drops a .meta.json sidecar, so:

python3 make_map.py deforestation_dndvi_m3p333_122p25          # by base name
python3 make_map.py data/mining_sirad_konawe.tif --basemap gray  # or a .tif path

Maps land in maps/. Tweak the layout in mapmaker.py and re-run instantly.

8 · Adapt it to your own area

Pick a location

Find coordinates by right-clicking a spot in Google Maps (it copies lat, lon), or from OpenStreetMap. Choose a --radius to match the feature: a few km for a single mine or town, 10–20 km for a flood plain or a fire.

Save a named preset

Add reusable places to sites.py and call them with --site:

SITES = {
  "my_area": {
    "label": "My study area",
    "lat": 1.234, "lon": 103.456, "radius_km": 8.0,
    "sentinel2_date": "2026-06-15",
    "sirad_periods": [("2024-01-01","2024-12-31"),
                      ("2025-01-01","2025-12-31"),
                      ("2026-01-01","2026-06-30")],
    "ndvi_pre":  ("2023-01-01","2023-12-31"),
    "ndvi_post": ("2026-01-01","2026-06-30"),
  },
}
python3 detect.py -s deforestation --site my_area --map

Add a brand-new scenario

Most scenarios are one dictionary entry in scenarios.py. To add, say, a moisture-change scenario using NDWI on a custom threshold you only add a registry entry — the generic optical-change engine does the rest:

"moisture": {
    "label": "Moisture change (NDWI)",
    "run": _optical("NDWI", "gain", 0.08, 0.20),  # index, direction, thresholds
    "radius": 10.0, "needs": "pre_post",
    "pre":  ("2023-01-01","2023-12-31"),
    "post": ("2025-01-01","2025-12-31"),
    "interpretation": "Greener = wetter.",
},

New spectral indices go in indices.py (a one-line normalised difference). Radar or bespoke methods get a small run_* function in scenarios.py.

9 · Reproducibility & data

  • Data: Copernicus Sentinel-1 (radar) and Sentinel-2 (optical), free and open, courtesy ESA, processed in Google Earth Engine.
  • AOI: a square centred on your coordinate (half-side = radius), axis-aligned in EPSG:4326.
  • Orbit auto-selection: for radar scenarios the tool checks both ascending and descending orbits and uses whichever has imagery in every period — coverage varies by location and year.
  • Outputs are regenerable, so results and maps are git-ignored; only code and this tutorial are versioned.

10 · Troubleshooting

SymptomFix
No Sentinel-2 scenes in pre/post windowWiden your --pre/--post dates or increase --radius. Very cloudy tropics may need a longer window.
No orbit covers all periods (radar)Enlarge the radius or shift the period dates; Sentinel-1 coverage is sparse in some years/areas.
Earth Engine auth errorsRe-run earthengine authenticate, or place a valid service-account key at scripts/config/ee-geodetic.json.
Map basemap missing / EPSG code unknownA stale external PROJ database (e.g. from an OTB install) is shadowing PROJ. The tool already clears PROJ_LIB; if it persists, unset PROJ_LIB in your shell.
Direct GeoTIFF download refusedVery large AOIs exceed Earth Engine's direct-download limit — reduce --radius, or use the --drive flag to export to Google Drive.
